# beacon-cloud 统一路线图与未完成项总文档

文档类型：路线图  
适用对象：开发 / 项目管理 / 答辩 / 排障  
验证基线：文档汇总整理  
关联模块：全局  
最后核对日期：2026-03-18


## 1. 这份文档解决什么问题

前面的文档大多是“按模块”拆开的：每个模块各自有风险、建议、P0/P1/P2。  
这会有一个实际问题：**单模块内的 P0，并不等于全局最先做**。

这份文档给出的不是“模块内优先级”，而是**跨模块统一执行顺序**：

1. 先做什么，能最快降低线上风险
2. 哪些模块必须一起改，不能拆开做
3. 哪些优化应该后置，否则会返工

---

## 2. 全局排序原则

统一优先级按下面 5 条判断：

1. 先做会导致“消息丢失 / 状态错乱 / 扣费异常 / 安全暴露”的项
2. 先做主链路一致性问题，再做外围体验优化
3. 先做公共契约和共享底座，再做依赖它们的业务模块
4. 先做“止血型改造”，再做“结构型重构”
5. 监控、测试、答辩增强要跟在主链路稳定之后，不要抢主优先级

---

## 3. 统一优先级结论

如果只给一句话版本，顺序建议是：

1. **安全与危险入口治理**
2. **主链路契约一致性**
3. **MQ / 网关可靠性**
4. **缓存与状态一致性**
5. **后台控制面持久化与权限治理**
6. **搜索 / 推送 / 监控完善**
7. **批次化、定时化、规模化演进**

---

## 4. 推荐执行顺序

## Phase 0：冻结边界，先止血

目标：先把最危险、最容易误操作、最容易造成不可逆损失的问题压住。

建议顺序：

1. `beacon-webmaster`
   - 明文配置、`test-kaptcha`、RBAC 缺失、Quartz 反射边界
2. `beacon-api`
   - 内部发送接口约束
3. `beacon-cache`
   - 密钥治理、调用权限治理、测试接口继续收口
4. `beacon-smsgateway`
   - 测试入口暴露、CMPP 凭证硬编码

为什么先做这一层：

1. 这些问题不一定天天触发，但一旦触发，影响是“越权、误调用、消息丢失、环境泄露”级别。
2. 这类改造通常不需要大改业务模型，收益高，返工少。

本阶段结束标准：

1. 高危测试入口全部可控
2. 核心密钥和账号不再裸奔
3. 后台权限边界清楚，不再是“登录即几乎全通”

### 4.0.1 问题本质

Phase 0 处理的不是“代码优不优雅”，而是“系统边界是否危险”。

这阶段的问题有 4 个共同点：

1. 不一定高频出现，但一旦触发就是事故级
2. 往往不需要改动核心业务模型，适合先止血
3. 会影响后续所有阶段的改造安全性
4. 如果不先做，后面的主链路和一致性改造都可能在不安全边界上继续堆复杂度

换句话说，Phase 0 的目标不是“把系统变先进”，而是**先让系统不那么容易出大错**。

### 4.0.2 执行边界

Phase 0 只做下面 4 类动作：

1. 收口测试入口、万能开关、明文敏感配置
2. 给高风险能力补最小授权边界
3. 让内部调用的最小安全前提明确且可控
4. 限制“任意执行”的能力，如反射任务、调试接口

Phase 0 暂时不做：

1. 主链路对象契约统一
2. MQ 可靠性重构
3. 缓存同步大改
4. 批次化、定时化、平台化演进

### 4.0.4 推荐执行顺序

虽然上面任务不少，但实际落地建议按下面顺序推进：

1. `P0-1` + `P0-2` + `P0-5`
   - 先收口 `beacon-webmaster` 登录和敏感配置
2. `P0-7`
   - 再把 `beacon-api` 的内部发送鉴权边界压住
3. `P0-9` + `P0-10` + `P0-11`
   - 然后治理 `beacon-cache` 这个共享底座的密钥和权限
4. `P0-12`
   - 最后收口网关凭证
5. `P0-4`
   - Quartz 反射边界建议在前面登录/RBAC 边界初步收口后做
6. `P0-14`
   - 最后补统一验收清单

### 4.0.5 总工期估算

按单人连续开发估算：

1. 最乐观：5 个工作日
2. 常规：6 到 8 个工作日
3. 如果涉及配置平台、权限模型和联调审批：8 到 10 个工作日

更保守的建议是按 **1.5 周** 规划 Phase 0。

### 4.0.6 验收清单

Phase 0 完成后，至少要逐项确认下面这些结果：

1. 生产类配置文件中不再直接出现真实 DB / MQ / CMPP 密码
2. `beacon-webmaster` 的 `test-kaptcha` 不能在生产样式环境下生效
3. `beacon-webmaster` 关键写接口不能再“登录即可调用”
4. `ScheduleInvokeServiceImpl` 不能再随意反射执行任意 Bean/Method
5. `beacon-cache`、`beacon-smsgateway` 的测试入口默认不可访问
6. `beacon-api` 的 internal token 规则在不同环境下是明确且可验证的
7. `beacon-cache` 的调用方权限矩阵与实际使用场景一致
8. `beacon-smsgateway` 的 CMPP 凭证不再硬编码在 Java 源码中
9. 日志里不再出现密码、验证码、裸 `System.out` 作为关键运行日志

### 4.0.7 这阶段最容易踩的坑

1. 只改文档不改启动失败策略
   - 结果是“看起来收口了”，其实生产仍能空配置启动
2. 只关测试入口，不关调用权限
   - 结果是调试面收了，但高危正式接口还是太宽
3. 一口气改完整个权限模型
   - 容易把 Phase 0 做成 Phase 4，工期失控
4. 不先出验收清单就开始改
   - 最后很难判断哪些边界真的收紧了

### 4.0.8 最小交付物

如果要把 Phase 0 当成一个独立迭代交付，建议最少交付这些东西：

1. 配置治理清单
2. 权限矩阵清单
3. 测试入口收口说明
4. Quartz 白名单/边界说明
5. 一份可执行的 Phase 0 验收 checklist

## Phase 1：修主链路一致性

目标：把一条短信从进入系统到发出、回执、更新状态这条主链路的关键“歧义点”全部收敛。

必须一起看的模块：

1. `beacon-common`
2. `beacon-api`
3. `beacon-strategy`
4. `beacon-smsgateway`

建议顺序：

1. `beacon-common`
   - 明确 `StandardSubmit` / `StandardReport` 契约
   - 补齐 `apiKey` 契约回归测试
2. `beacon-api`
   - internal / external 入口字段一致性
   - `fee` 生成规则一致性
3. `beacon-strategy`
   - `dirtyword` 语义统一
4. `beacon-smsgateway`
   - 回执关联字段严格按统一契约走

为什么这一层必须一起做：

1. 这里是“对象契约 + 业务语义”问题，单改一个模块很容易把错误挪到下游。
2. `apiKey`、`fee`、失败回调、敏感词拦截，这些都跨越多个服务。

本阶段结束标准：

1. internal / external 发送行为一致
2. 敏感词、扣费、失败回调语义稳定
3. 主链路对象字段不再靠猜

---

## Phase 2：修可靠性，不让消息丢

目标：保证“消息发没发出去、回执有没有对上、状态有没有更新”这三件事可控。

必须一起看的模块：

1. `beacon-smsgateway`
2. `beacon-search`
3. `beacon-push`

建议顺序：

1. `beacon-smsgateway`
   - Ack 与发送结果绑定
   - 提交/回执关联状态治理
   - Netty 重连模型治理
2. `beacon-search`
   - `ThreadLocal` 生命周期
   - 更新补偿闭环
   - 写入幂等策略
3. `beacon-push`
   - `RestTemplate` 超时
   - 回调 URL 规范化
   - 重试交换机可靠性

为什么这一层排在 Phase 1 后：

1. 如果对象契约还没稳，先做可靠性改造会把错误稳定地放大。
2. 等主链路语义稳定后，再做消息可靠性，收益更大。

本阶段结束标准：

1. 发不出去时不会直接 Ack 丢消息
2. 回执对不上时有清晰降级与监控
3. 搜索状态更新、回调重试不再出现“静默失败”

---

## Phase 3：治理共享底座与缓存一致性

目标：把“公共依赖”和“运行态配置一致性”拉到可长期维护的水平。

必须一起看的模块：

1. `beacon-cache`
2. `beacon-common`
3. `beacon-webmaster`
4. `beacon-webmaster/src/docs/21_mysql_redis_sync_fix_guide.md` 对应方案

建议顺序：

1. `beacon-cache`
   - 序列化升级
   - typed API
   - `sinterstr` 原子化
2. `beacon-common`
   - 契约测试
   - 进程内状态外部化方向明确
3. `beacon-webmaster`
   - 缓存同步统一门面
   - 运行时同步补齐域接入
4. 缓存同步专题
   - Boot Reconcile
   - Manual Rebuild
   - 验收脚本

为什么这一层放在这里：

1. 主链路不稳时，缓存同步体系做得再完整，也会同步错数据。
2. 等 Phase 1/2 稳住后，再做缓存一致性，才能减少返工。

本阶段结束标准：

1. 配置改动能自动同步
2. Redis key 体系和命名空间规则稳定
3. 启动校准和手工重建具备可操作方案

---

## Phase 4：治理后台控制面

目标：把后台从“能用”提升到“能长期维护”。

重点模块：

1. `beacon-webmaster`
2. `beacon-test`

建议顺序：

1. `beacon-webmaster`
   - `Acount/Phase/Legacy` 持久化替换
   - 分页下推 DB
   - Feign 契约强类型化
2. `beacon-test`
   - 工具职责收敛
   - JUnit 统一
   - 断言与日志规范化

为什么这里后置：

1. 后台控制面很重要，但比“主链路正确发短信”优先级还是低一层。
2. 它依赖前面几层的契约、缓存、可靠性都先稳定。

本阶段结束标准：

1. 后台不再大量依赖内存数据结构
2. 大部分管理列表不再查全量再内存分页
3. 联调工具和生产逻辑边界更清楚

---

## Phase 5：补监控、补测试、补验收

目标：把前面改完的东西真正变成“可持续维护”。

重点模块：

1. `beacon-monitor`
2. 所有模块的测试
3. 本文中的未完成项矩阵维护

建议顺序：

1. 给每个主链路模块补最小回归测试
2. 补监控指标和告警
3. 统一验收清单和答辩演示脚本

这一步不要提前做的原因：

1. 主逻辑没稳定前，测试和监控会反复跟着重写。
2. 等结构大致稳定后再补，投入产出比更高。

---

## Phase 6：最后再做规模化演进

目标：从“单条发送系统”演进到“任务化、批次化、定时化”的平台。

重点参考：

1. 本文第 8 节的演进方向摘要
2. 系统总览与主链路总文档

建议顺序：

1. 批次受理
2. 拆分任务
3. 定时发送
4. 模板中心
5. 网关状态外部化

为什么必须最后做：

1. 这是放大规模，不是先修正确性。
2. 如果现在主链路和缓存一致性都还没稳，越做大越难收。

---

## 5. 一句话优先级

如果你要一个最实用的“先后顺序”，我建议就是：

1. **先安全**
2. **再统一主链路契约**
3. **再修 MQ / 网关可靠性**
4. **再补缓存一致性**
5. **再重构后台控制面**
6. **最后补监控测试和规模化能力**

## 8. 演进方向摘要

如果系统从“单条发送平台”继续往前走，最值得投入的 4 个方向是：

1. 批次化群发
   - 从“单请求单号码”升级到“单任务多号码”
2. 定时 / 分时发送
   - 支持时间窗口、延迟投递、定时触发
3. 模板化与变量化
   - 提高合规性，降低内容拼装成本
4. 状态与任务外部化
   - 把本地临时状态迁移为可恢复、可审计的共享状态

一句话判断：

1. 这些都应该排在主链路、安全边界和一致性治理之后
2. 否则只是把当前问题放大到更大规模
