## 1. 这份文档解决什么问题

`02_统一重构优先级路线图.md` 已经明确了：整个项目的第一优先级是 `Phase 0：冻结边界，先止血`。  
但路线图里的 P0 还是偏“总览”，适合排优先级，不够适合直接分派执行。

这份文档的目标是把 P0 变成一份**可以直接推进的执行稿**，明确：

1. Phase 0 到底做什么，不做什么。
2. 每个 P0 任务的当前风险、改造目标、实施动作、验收口径。
3. 哪些任务可以并行，哪些任务需要前置依赖。
4. 这一轮最小交付物是什么，做到什么程度就算 Phase 0 结束。

一句话概括：**这不是“继续扩功能”，而是先把最危险的边界收紧，让系统不那么容易出事故。**

---

## 2. Phase 0 的范围边界

## 2.1 本阶段只做 4 类事情

1. 收口测试入口、万能开关、明文敏感配置。
2. 给高风险能力补最小授权边界。
3. 明确内部调用的最小安全前提，并让缺失配置时行为确定。
4. 限制“任意执行”能力，例如反射调度、调试接口、硬编码外部凭证。

## 2.2 本阶段明确不做

1. 不统一主链路对象契约，不改 `StandardSubmit` / `StandardReport` 语义。
2. 不做 MQ 可靠性大改，不重写发送确认和回执闭环。
3. 不做缓存同步体系重构，不做 typed API 全量切换。
4. 不做后台持久化重构，不把内存模块改成数据库。
5. 不做批次化、任务化、定时化、平台化演进。
6. 不顺手把所有历史代码“顺便重构干净”。

## 2.3 Phase 0 的判断标准

只有满足下面任意一条，才应该进入 Phase 0：

1. 一旦触发就可能造成越权、误调用、明文泄露、调度滥用、不可逆错误。
2. 不需要改核心业务模型，只需收紧边界就能显著降风险。
3. 如果现在不做，后续 Phase 1/2 会在不安全前提上继续堆复杂度。

## 3. 当前高风险现状摘要

下面是本次 P0 直接对应的仓库现状信号，供执行前统一认知。

| 模块 | 当前信号 | 事故类型 |
| --- | --- | --- |
| `beacon-webmaster` | `application.yml` 直接包含 DB/Nacos 地址与账号密码；`system.test-kaptcha: "1111"` 常驻主配置；`ShiroConfig` 以 `/** -> authc` 为主；`ShiroRealm#doGetAuthorizationInfo()` 返回 `null`；`ScheduleInvokeServiceImpl` 允许按 DB 中的 `beanName + methodName` 反射执行；`SmsUserController` 日志会打印 `userDTO`、验证码输入/期望值 | 配置泄露、登录面绕过、登录后越权、任意执行、敏感日志泄露 |
| `beacon-api` | `SmsController` 中 `internal.sms.token` 为空时不做校验，环境边界不清晰 | 内部发送鉴权失效 |
| `beacon-cache` | 已接入第一轮签名鉴权与权限模型，但 `bootstrap.yml` 仍承载 `caller-secrets` / `caller-permissions`；`TestController` 虽默认关闭，但仍在主应用路径中 | 共享底座密钥泄露、权限漂移、测试接口再次暴露 |
| `beacon-smsgateway` | `NettyStartCMPP` 直接硬编码 `host/port/serviceId/pwd`；`/test` 暴露在线上包 | 运营商凭证泄露、测试入口误触发 |

要特别注意：`beacon-cache` 和 `beacon-webmaster` 并不是“完全没做防护”，而是**已经有了第一步治理，但边界还没彻底收紧**。Phase 0 的目标是把这层“半收口”变成“可验证的收口”。

## 4. Phase 0 总体完成标准

Phase 0 完成后，至少要同时满足下面 8 条：

1. 生产类配置文件中不再直接出现真实 DB / MQ / CMPP / 调用密钥。
2. `beacon-webmaster` 的 `test-kaptcha` 不能在生产样式环境生效。
3. `beacon-webmaster` 关键写接口不再是“登录即可调用”。
4. 调度任务不能再按任意 `Bean/Method` 反射执行。
5. `beacon-api`、`beacon-cache` 的测试入口默认不可访问。
6. `beacon-api` 的 `X-Internal-Token` 规则在不同环境下是明确且可验证的。
7. `beacon-cache` 的调用方权限矩阵与真实调用场景一致。
8. `beacon-smsgateway` 的 CMPP 凭证不再硬编码在 Java 源码中。
9. 运行日志中不再打印密码、验证码、原始敏感 DTO、裸 `System.out` 作为关键日志手段。

## 6. 任务明细

## 6.1 `beacon-webmaster`

### P0-1 明文敏感配置迁出仓库

当前现状：

1. `beacon-webmaster/src/main/resources/application.yml` 直接包含：
   - `spring.datasource.url`
   - `spring.datasource.username`
   - `spring.datasource.password`
   - `spring.cloud.nacos.discovery.server-addr`
   - `spring.cloud.nacos.config.server-addr`
2. 主配置文件中还包含 `spring.profiles.active: dev` 这类环境相关默认值。
3. `internal.balance.token` 目前允许空值，部署边界不够显式。

风险说明：

1. 一旦仓库、制品、日志或截图泄露，配置会被直接带出。
2. 多环境配置无法靠来源隔离，只能靠“人不犯错”。
3. 后续联调容易把开发配置误带入测试/生产。

改造目标：

1. 仓库中只保留非敏感默认值和配置说明，不保留真实账号密码。
2. 非本地环境必须从 Nacos、环境变量或密钥中心注入配置。
3. 缺关键配置时，服务在启动阶段明确失败，而不是带着空值运行。

详细动作：

1. 将数据库、Nacos、内部 token 等配置改为占位符形式。
2. 新增或补齐部署说明，写清配置来源、配置键、责任方。
3. 区分“仓库内样例值”和“运行时真实值”，避免样例值被误认为线上值。
4. 非本地环境加 fail-fast 校验：
   - 缺 DB 连接串时拒绝启动。
   - 缺 Nacos 地址时拒绝启动。
   - 需要 token 的场景下空值拒绝启动。
5. 清点所有 `application*.yml` / `bootstrap*.yml` 是否还有同类明文。

验收标准：

1. 仓库中不再提交真实数据库密码、Nacos 地址、真实 token。
2. 启动文档能明确说明配置从哪里来。
3. 在非本地环境模拟缺失配置时，服务启动失败且失败原因明确。

---

### P0-2 `system.test-kaptcha` 收口

当前现状：

1. `application.yml` 中固定存在 `system.test-kaptcha: "1111"`。
2. `SmsUserController` 中只要输入验证码等于该值，就可以跳过真实验证码校验。

风险说明：

1. 这是典型的“万能开关”，如果误进入生产会直接削弱登录面防护。
2. 该类开关最危险的地方不是经常被用，而是**很难从外部观察出它还开着**。

改造目标：

1. `test-kaptcha` 只能在 `local/dev` 这类明确受控环境生效。
2. 生产样式环境中即便误配该值，也不能悄悄生效。

详细动作：

1. 将 `system.test-kaptcha` 从主配置移出，改为仅在 `local` 或 `dev` profile 下声明。
2. 在启动阶段增加环境校验：
   - 若当前环境不是 `local/dev`，但配置了 `test-kaptcha`，则启动失败。
3. 登录逻辑中保留清晰日志，但不要记录验证码原值。
4. 在部署手册中明确：
   - 本地联调用途
   - 非本地禁用规则
   - 启用前提和风险说明

验收标准：

1. 非 `local/dev` 环境无法通过万能验证码登录。
2. 非本地环境一旦配置该开关，启动或运行期会明确失败。
3. 回归时真实验证码路径仍可正常使用。

### P0-3 补最小 RBAC 授权边界

当前现状：

1. `ShiroConfig` 目前以 `/** -> authc` 为主。
2. `ShiroRealm#doGetAuthorizationInfo()` 返回 `null`，说明授权链基本未真正生效。
3. 高危管理接口虽然不是匿名可调，但本质上仍接近“登录即可调用”。

风险说明：

1. 登录态不等于有权限。
2. 一旦后台账号被滥用或越权使用，改配置、调任务、删数据的风险都被放大。

改造目标：

1. 先把最危险的写接口收口到“有权限才能调用”。
2. 先做最小权限闭环，不在 Phase 0 里一口气重写完整权限模型。

Phase 0 只收口这类接口：

1. 调度任务保存、修改、删除、暂停、恢复、立即执行。
2. 用户、角色、菜单、客户、通道、账号等高危写操作。
3. 运行期开关、同步、手工触发、批量变更类入口。

详细动作：

1. 实现 `doGetAuthorizationInfo()`，让角色和权限从现有用户体系真正装载到 Shiro。
2. 给高危控制器补最小权限注解或统一权限拦截。
3. 明确最小权限命名规则，例如：
   - `schedule:job:run`
   - `schedule:job:save`
   - `sys:user:update`
   - `sys:role:grant`
4. 对暂时无法完整梳理的历史接口，至少先按“读/写/运维”三级收口。
5. 增加越权访问的拒绝日志，但不回传底层异常。

验收标准：

1. 关键写接口能区分“已登录但无权限”和“有权限”两类用户。
2. 越权访问返回拒绝，而不是业务异常或静默成功。
3. 至少覆盖调度、角色授权、后台关键配置写操作。

注意事项：

1. Phase 0 不要求把所有页面按钮级权限都补齐。
2. 不要求现在切换密码算法或完成权限模型大重构。

### P0-4 Quartz 反射边界收口

当前现状：

1. `ScheduleInvokeServiceImpl` 通过数据库中的 `beanName`、`methodName`、`params` 找 Bean 后直接反射执行。
2. `ScheduleJobController` 捕获异常后直接 `Result.error(e.getMessage())` 返回前端。

风险说明：

1. 这是典型的高危运维入口。
2. 任意 Bean/Method 反射执行，一旦配置错误或越权调用，风险上限非常高。
3. 异常原文回前端会泄露内部实现细节。

改造目标：

1. 从“任意执行”改成“白名单执行”。
2. 把异常暴露从“底层异常原文”改成“统一错误码/错误描述”。

详细动作：

1. 建立 `jobType -> handler` 白名单注册表。
2. 数据库层或保存接口层不再接受任意 `beanName + methodName` 组合。
3. 参数从自由字符串，收紧到受控格式：
   - JSON 结构
   - 固定枚举值
   - 明确 schema
4. 对保存、修改、立即执行入口同步做白名单校验，而不是只在运行时拦截。
5. `ScheduleJobController` 统一返回外层错误信息，不把栈、类名、方法名直接透出。
6. 调度日志保留：
   - `jobId`
   - `jobType`
   - 参数摘要
   - 执行结果
   - 耗时

验收标准：

1. 非白名单任务不能保存、不能运行。
2. 白名单任务执行失败时，前端拿到统一错误信息，不直接看到内部异常原文。
3. 调度日志能用于定位，但不会直接暴露敏感堆栈给前端。

前置依赖：

1. 建议先完成 `P0-3` 的最小 RBAC 收口。

### P0-5 登录失败日志脱敏

当前现状：

1. `SmsUserController` 中失败日志会打印 `userDTO`。
2. 验证码错误时会打印 `input` 和 `expected`。

风险说明：

1. `userDTO` 可能包含用户名、密码、验证码等敏感字段。
2. 打印“期望验证码”相当于把校验答案写入日志。

改造目标：

1. 登录审计可用，但日志中不出现密码、验证码、完整 DTO。

详细动作：

1. 删除 `userDTO={}` 这类整对象日志。
2. 删除 `input/expected` 这类验证码原值日志。
3. 统一记录以下非敏感审计字段：
   - 用户名
   - 请求 IP
   - 失败类型（参数错误 / 验证码错误 / 认证失败）
   - 时间
4. 如需要排障，单独记录 traceId，不把敏感字段直接写日志。

验收标准：

1. 登录相关日志不再出现密码、验证码、完整请求 DTO。
2. 仍能通过日志区分不同失败类型。

## 6.2 `beacon-api`

### P0-7 内部发送 token 前提明确化

当前现状：

1. `SmsController` 通过 `@Value("${internal.sms.token:}")` 读取 token。
2. 只有当配置非空时才校验 `X-Internal-Token`。
3. 也就是说：如果 token 为空，内部发送接口会落入“默认不校验”的隐式状态。

风险说明：

1. 这是最典型的“配置缺失时默认放行”。
2. 一旦被误以为“系统已经接了 token”，实际上却没有任何鉴权。

改造目标：

1. 明确不同环境的行为规则，不允许处于模糊状态。

建议规则：

1. `local/dev`：可以允许 token 为空，但必须显式声明是开发模式。
2. `test/stage/prod`：token 必须配置，否则拒绝启动。
3. 任意非开发环境：token 错误时返回明确鉴权失败，不进入业务逻辑。

详细动作：

1. 将 `internal.sms.token` 改为 `@ConfigurationProperties + @Validated` 管理。
2. 增加环境策略开关，例如：
   - 是否要求 internal auth
   - 哪些 profile 允许空值
3. 启动阶段做 fail-fast：
   - 非开发环境 token 缺失则启动失败
4. 保留接口层校验：
   - 缺 header
   - token 不匹配
   - 都返回统一错误
5. 在说明文档里明确：
   - Header 名称：`X-Internal-Token`
   - 谁负责配置
   - 哪些环境可空

验收标准：

1. 非开发环境 token 缺失时，行为是确定的，不是“自动放行”。
2. 错误 token 请求不会进入业务发送逻辑。
3. 环境规则可以被测试和验证。

## 6.3 `beacon-cache`

### P0-9 `callerSecrets` 脱敏与外置

当前现状：

1. `beacon-cache/src/main/resources/bootstrap.yml` 中已有 `cache.security.caller-secrets`。
2. 当前仓库中看起来是示例值，但真正的问题是：**密钥来源模式仍然允许落在仓库配置里**。
3. 调用方模块也各自有 `cache.client.auth.secret`。

风险说明：

1. `beacon-cache` 是共享底座，一旦密钥管理做错，会影响多个调用方。
2. 这类密钥与 DB 密码不同，它会扩散到多个模块，泄露面更广。

改造目标：

1. 服务端和调用端都不在仓库里提交真实 secret。
2. 配置来源统一，避免一部分模块走环境变量，一部分模块走明文。

详细动作：

1. 服务端只保留 caller 名称和配置说明，不保留真实 secret。
2. 调用端 `cache.client.auth.secret` 改为环境变量、Nacos 密文或密钥中心注入。
3. 统一密钥命名规范，例如：
   - `CACHE_AUTH_SECRET_BEACON_API`
   - `CACHE_AUTH_SECRET_BEACON_STRATEGY`
4. 如短期无法做密钥中心，至少先完成：
   - 仓库去真实值
   - 启动强校验
   - 配置来源统一说明
5. 补发布说明，明确服务端和调用端必须同步更新。

验收标准：

1. 仓库中不再出现真实 cache 调用密钥。
2. 服务端与调用端配置来源一致。
3. 在开启鉴权时，缺密钥会明确失败，不带着空值启动。

### P0-10 权限矩阵最小化

当前现状：

1. `beacon-cache` 已经有 `caller-permissions`。
2. 当前基线权限如下：
   - `beacon-api`: `READ`
   - `beacon-strategy`: `READ` `WRITE`
   - `beacon-monitor`: `READ` `KEYS`
   - `beacon-webmaster`: `WRITE`
   - `beacon-smsgateway`: `READ`
   - `beacon-test`: `READ` `WRITE` `TEST`
3. 这已经是第一版权限模型，但还需要核对“是否真的最小化”。

风险说明：

1. 权限矩阵最怕“先给宽一点，后面再收”，最后很难收回。
2. 共享底座的权限漂移，等于把多个业务模块的风险放大。

改造目标：

1. 每个调用方只拿到自己真实需要的权限。
2. 读、写、扫描、测试权限彻底分开。

详细动作：

1. 逐个调用方梳理实际接口调用面。
2. 对每个调用方形成“接口 -> 权限”映射。
3. 明确哪些模块绝不应该拥有：
   - `WRITE`
   - `KEYS`
   - `TEST`
4. 对 `beacon-webmaster` 这类管理端要特别确认：
   - 是只写不读
   - 还是部分场景确实需要读
   - 不能凭感觉给权限
5. 对 `KEYS` 权限额外加一层审计和发布确认。

建议产出一个最小权限矩阵表：

| 调用方 | 允许权限 | 禁止权限 | 备注 |
| --- | --- | --- | --- |
| `beacon-api` | `READ` | `WRITE` `KEYS` `TEST` | 只做鉴权、模板、余额等读取 |
| `beacon-strategy` | `READ` `WRITE` | `KEYS` `TEST` | 路由、扣费、黑名单等写操作 |
| `beacon-monitor` | `READ` `KEYS` | `WRITE` `TEST` | 监控巡检允许扫描白名单 pattern |
| `beacon-webmaster` | 待按真实调用核验 | 不默认给 `KEYS` `TEST` | 不确定项要靠代码调用面确认 |
| `beacon-smsgateway` | `READ` | `WRITE` `KEYS` `TEST` | 读取回调相关配置 |
| `beacon-test` | `READ` `WRITE` `TEST` | 生产不可启用 | 仅受控测试使用 |

验收标准：

1. `caller-permissions` 与真实调用场景一致。
2. 未被业务证明需要的权限，一律不授予。
3. 用错误权限访问接口时，能稳定返回拒绝。

### P0-11 测试接口继续收口

当前现状：

1. `TestController` 已有 `@ConditionalOnProperty(... test-api-enabled=true)`。
2. 这说明已经做了第一轮收口。
3. 但它仍在主应用内，仍需明确“默认关闭、仅受控开启、最好最终迁出主包”的策略。

风险说明：

1. 只要还在主应用里，未来就有被误开启或被误依赖的可能。
2. 测试接口一旦被业务方“临时借用”，后续会变成长期隐患。

改造目标：

1. 默认关闭并且文档可验证。
2. 即使开启，也只能在受控场景使用。

详细动作：

1. 保持 `cache.security.test-api-enabled=false` 作为默认值。
2. 明确只有：
   - 本地联调
   - 专项排障
   - 受控测试环境
   才允许临时开启。
3. 开启时仍要求调用方具备 `TEST` 权限。
4. 在文档中写清楚：
   - 开启前提
   - 使用窗口
   - 关闭要求
5. Phase 0 先完成默认关闭和权限收口；后续阶段再考虑彻底迁出主应用。

验收标准：

1. 默认配置下 `/test/**` 返回 `404`。
2. 未具备 `TEST` 权限的调用方即便知道路径也无法访问。

## 6.4 `beacon-smsgateway`

### P0-12 CMPP 凭证与连接参数外置

当前现状：

1. `NettyStartCMPP` 直接在代码中定义：
   - `host`
   - `port`
   - `serviceId`
   - `pwd`
2. 当前是静态字段 + `@Bean(initMethod = "start")` 创建客户端。

风险说明：

1. 账号、密码一旦入库，就是直接泄露面。
2. 环境切换需要改代码，运维成本高且容易误发布。
3. 后续多通道扩展会被当前硬编码结构卡住。

改造目标：

1. 凭证和连接参数全部从配置注入，不再写死在 Java 源码。
2. 非本地环境缺关键参数时，明确失败。

详细动作：

1. 引入 `@ConfigurationProperties(prefix = "cmpp")` 或通道化配置模型。
2. 通过 Nacos/环境变量注入：
   - host
   - port
   - serviceId
   - password
3. 启动阶段校验：
   - host 非空
   - port 合法
   - serviceId 非空
   - password 非空
4. 日志中禁止打印明文密码。
5. 部署说明中写清配置来源和密钥归属。

验收标准：

1. Java 源码中不再硬编码 CMPP 凭证。
2. 非本地环境缺失参数时明确启动失败。
3. 切换环境时不需要改源码。

## 6.5 全局 `P0-14`

### Phase 0 验收脚本与回归清单

当前现状：

1. 路线图已经有验收方向，但还缺一份独立、可执行、可打勾的清单。
2. 如果没有统一回归，很容易出现“代码改了，但边界其实没收紧”。

改造目标：

1. 给本阶段形成统一的检查清单和发布后核验动作。

本次最小要求：

1. 有一份配置治理清单。
2. 有一份权限矩阵清单。
3. 有一份测试入口暴露矩阵。
4. 有一份 Quartz 白名单/边界说明。
5. 有一份发布后检查步骤。

建议回归动作：

1. 配置检查
   - 代码仓库中不再出现真实密码、真实 secret、真实 CMPP 凭证。
2. 登录检查
   - 非本地环境 `test-kaptcha` 不生效。
3. 权限检查
   - 登录无权限用户访问高危写接口时被拒绝。
4. 调度检查
   - 非白名单任务不能保存/执行。
5. API 检查
   - `/api/test` 不可访问。
   - `X-Internal-Token` 错误时内部发送被拒绝。
6. Cache 检查
   - `/test/**` 默认不可访问。
   - 无签名请求访问 `/cache/**` 返回拒绝。
   - 非授权 caller 访问 `KEYS` 或写接口被拒绝。
7. Gateway 检查
   - 代码中不再硬编码 CMPP 凭证。

建议把验收结果落成表格：

| 检查项 | 环境 | 结果 | 证据 | 负责人 |
| --- | --- | --- | --- | --- |
| 非本地 `test-kaptcha` 禁用 | dev-like / test / prod-like | Pass / Fail | 截图或接口响应 | 待填写 |
| `/api/test` 不可访问 | 指定环境 | Pass / Fail | 响应码 | 待填写 |
| Quartz 非白名单拒绝 | 指定环境 | Pass / Fail | 请求与响应 | 待填写 |

## 7. 发布、回滚与依赖管理

## 7.1 发布顺序建议

1. 先发配置承接能力，再切配置源。
2. 先发调用方支持，再开启服务端强校验。
3. 先让默认值更安全，再逐步关掉旧入口。

具体建议：

1. `beacon-cache`
   - 先让调用方支持外部 secret
   - 再切服务端 secret 来源
2. `beacon-api`
   - 先明确 token 规则
   - 再在非开发环境启用强校验
3. `beacon-webmaster`
   - 先完成登录面与日志脱敏
   - 再上 RBAC 最小收口
4. `beacon-smsgateway`
   - 先接配置注入
   - 再移除代码硬编码

## 7.2 回滚原则

1. 可以回滚“配置来源”，不要回滚“安全边界”。
2. 能通过开关临时降级的，只作为紧急手段，不作为长期方案。
3. 如果回滚会重新暴露高危入口，必须记录风险并限定时间窗口。

## 7.3 本阶段最容易踩的坑

1. 只改文档，不做启动 fail-fast。
2. 只关测试接口，不关正式高危接口权限。
3. 一口气把权限模型全部重做，导致工期失控。
4. 没有统一验收清单就开始改，最后无法证明边界真的收紧。
5. 以为“仓库里现在是示例值”就代表配置治理已经结束。

## 8. 最小交付物

如果要把 Phase 0 当成一个单独迭代交付，最少应该交付这些东西：

1. 一份配置治理清单
   - 哪些配置迁出仓库
   - 配置来源是什么
   - 哪些环境必须 fail-fast
2. 一份最小权限矩阵

   - cache caller 权限矩阵

   - `beacon-api`
   - `beacon-cache`
   - `beacon-smsgateway`
4. 一份 Quartz 白名单/边界说明
5. 一份可执行的 Phase 0 验收 checklist

## 9. 一句话执行结论

Phase 0 的正确做法不是“先把代码重构漂亮”，而是先把最危险的东西关起来。  
这轮最该优先完成的是：

1. `beacon-webmaster` 登录面与权限边界
2. `beacon-api` 内部发送前提与测试入口
3. `beacon-cache` 共享底座密钥与权限矩阵
4. `beacon-smsgateway` 凭证外置与测试入口
5. Quartz 从“任意执行”改成“白名单执行”

只有这层止血完成后，后面的主链路一致性、MQ 可靠性、缓存一致性重构才值得继续往下做。
