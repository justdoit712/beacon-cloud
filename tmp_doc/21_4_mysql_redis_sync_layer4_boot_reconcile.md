# 第四层：启动校准（执行文档）

## 0. 目标

第四层用于在服务启动后做一次有限域校准，解决云端 Redis 残留和本地重启后脏缓存问题。

本层必须复用第三层重建能力，不能另起一套逻辑。

### 0.1 文档引用规则

1. 本层沿用第一层冻结后的范围口径、命名空间口径、主线域档案和余额前置条件。
2. boot 默认域选择必须以总览矩阵和第三层支持域为准。
3. 若本层描述与第一层或总览矩阵冲突，以第一层和总览文档为准。

### 0.2 评审入口检查

第四层进入设计或代码评审前，至少先检查：

1. 是否直接复用了第三层重建核心逻辑。
2. boot 默认域是否仍与总览矩阵一致。
3. 第二层未完成前是否避免默认纳入 `client_balance`。
4. 是否没有额外设计第二套 loader 或第二套重建规则。

---

## 1. 本层前置条件

1. 第一层公共约束已冻结。
2. 第二层主线域运行时同步稳定。
3. 第三层手工重建已经可用。

---

## 2. 本层范围

### 2.1 默认纳入域

1. `client_business`
2. `channel`
3. `client_channel`

### 2.2 条件纳入域

1. `client_balance`
   - 仅在第二层完成后才允许开启

### 2.3 不纳入域

1. `client_sign`
2. `client_template`
3. `black`
4. `dirty_word`
5. `transfer`

---

## 3. 本层固定决策

### 3.1 复用决策

1. 启动校准直接调用第三层 `CacheRebuildService`。
2. 不允许再设计第二套 loader 或重建规则。

### 3.2 域选择决策

候选域必须同时满足：

1. `boot.enabled=true`
2. 域已注册 loader
3. 域允许 boot rebuild
4. 域属于当前范围

### 3.3 失败处理决策

1. 启动校准失败不阻塞 Spring 启动成功。
2. 但必须输出 error 日志和汇总报告。
3. 单域失败不影响后续域继续执行。

### 3.4 并发保护决策

1. 与第三层复用同一套域级锁。
2. 多实例不能并发重建同一域。

---

## 4. 本层新增能力

新增组件：

1. `CacheBootReconcileRunner implements ApplicationRunner`

建议配置：

1. `sync.boot.enabled`
2. `sync.boot.domains`

---

## 5. 执行流程

1. 服务启动后检查 `sync.boot.enabled`。
2. 解析待校准域列表。
3. 按规则过滤出真正允许执行的域。
4. 逐域调用第三层重建服务。
5. 汇总每个域的结果。
6. 输出日志和统计。

---

## 6. 日志与观测要求

每个域至少输出：

1. 域名
2. 开始时间
3. 结束时间
4. 耗时
5. 成功数
6. 失败数
7. 失败 key

启动结束后还要输出一次汇总结果。

---

## 7. 测试要求

1. 启动校准只跑允许的域。
2. 多实例场景不会并发重建同一域。
3. 单域失败不会导致整个应用启动失败。
4. `client_balance` 在第二层完成前不会被默认带上。
5. 启动校准复用第三层逻辑，没有第二套分叉实现。

---

## 8. 验收标准

1. 启动后有限域自动校准可执行。
2. 失败可以定位到具体域。
3. 不会因校准失败阻塞服务启动。
4. 范围与第三层支持域保持一致，不会误带 legacy 域。
